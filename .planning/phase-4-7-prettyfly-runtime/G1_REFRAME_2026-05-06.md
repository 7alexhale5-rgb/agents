# G1 baseline reframe

## §0 Why this doc exists, not a script change

Phase 4.7.0 cleared with `g1-baseline-capture.sh` shipped under a metric contract that — surfaced by tonight's empirical probe and the four-perspective swarm — cannot produce useful baseline data as written. The qualifying-night gate (`≥5 sessions/night WHERE source='slack'`) is structurally unmeetable because Hermes coalesces sessions by `(channel × user × day)` per `gateway/config.py:229-231` (`mode='both'`, `idle_minutes=1440`, `at_hour=4`). One operator DMing one bot produces ≤1 session row per day regardless of turn count.

The first-principles reframe (atom-level finding 1+2) and the architecture lens (finding 1) converged on a deeper observation: per-turn metrics don't fix this — they relocate the N-insufficiency problem. The skeptic flagged finding 4: I was conflating two orthogonal axes (qualifying-night data-volume gate vs cutover-quality gate) into one threshold.

This doc rewrites the G1 contract to fix the conflation. It ships as a design artifact for Codex re-review. The script changes that follow this doc are gated on Codex returning ≤2 critical findings.

## §1 Contract decision A — separate quality from operational signal

The 4.7.5 cutover gate compares Hermes baseline against PF Runtime on five metrics. Tonight's reframe splits those five into two independent surfaces:

**Quality surface** (synthetic, calendar-independent, statistically dense):

- Promptfoo Wilson lower-CI on the 30-question golden set, N=150 via `--repeat 5`
- Ragas answer_relevance score
- Both run nightly via `scripts/g1-baseline-capture.sh` invoking the eval pipeline against `hermes/profiles/personal/eval/promptfoo.yaml`
- Independent of operator DM volume, Hermes session-coalescing, or live-traffic patterns
- This is the gate that 4.7.5's quality threshold (`Hermes Ragas ±0.02`) calibrates against

**Operational surface** (live, sample-size-disclosed, NOT a quality gate):

- Per-turn cost (denominator TBD — see §5)
- Per-turn latency (definition TBD — see §6)
- Trace volume per 24h
- Captured nightly when traffic exists, sample size disclosed in every row
- Used for 4.7.5's cost-tolerance and latency-tolerance gates with **explicit confidence bands** rather than fixed-tolerance ratios

The quality surface drives go/no-go decisions. The operational surface informs tolerance bands at cutover. They never share a threshold.

## §2 Contract decision B — qualifying-night redefinition

The original gate (`≥5 sessions/night`) is replaced with a quality-anchored definition that reflects what G1 is actually measuring:

> **A night qualifies for G1 if** the Promptfoo eval ran cleanly (`results.stats.errors == 0`) AND emitted ≥30 graded answers (i.e., the full golden set ran at least once) AND the Ragas score was computed without divide-by-zero.

The live-traffic operational signal does NOT factor into qualifying. If traffic exists, capture it; if not, leave the column sparse. The "7 nights of qualifying data" requirement applies to synthetic quality — that's deterministic and operator-independent.

Implementation note: the existing `MIN_SESSIONS_PER_NIGHT=5` gate in `scripts/g1-baseline-capture.sh` **will be removed in the §8 implementation commit, gated on Codex re-review of this doc returning ≤2 critical findings**. As of this commit, the gate is still active in the script — that is the intentional pre-Codex state. The §8 commit replaces the gate with `errors==0` + `len(graded_answers)>=30` from the Promptfoo JSON output.

## §3 Contract decision C — 4.7.5 cutover gate redefinition

Phase 4.7's PLAN.md §11 lists 5 cutover gates. After this reframe they become:

| Gate                  | Surface     | Threshold                                                                                                 |
| --------------------- | ----------- | --------------------------------------------------------------------------------------------------------- |
| 1. Quality            | Synthetic   | PF Runtime Ragas ≥ Hermes Ragas − 0.02 (per existing 0.85/0.80 floor in HERMES_BASELINE.md decision rule) |
| 2. Wilson floor       | Synthetic   | PF Runtime Wilson lower-CI ≥ 0.80 absolute                                                                |
| 3. Cost tolerance     | Operational | PF Runtime per-turn cost p50/p95 within Hermes's 90% confidence band (NOT a fixed ±10%)                   |
| 4. Latency tolerance  | Operational | PF Runtime per-turn latency p95 within Hermes's 90% confidence band (NOT a fixed ±30%)                    |
| 5. Real-job execution | Live        | ≥1 successful real-job round-trip per profile (Marc/sportsbook/lawdbot/yeh-ops) — unchanged               |

Gates 1+2 carry hard quality thresholds. Gates 3+4 are confidence-band-relative — the bands widen automatically when N is low, which is the honest disclosure of statistical uncertainty.

## §4 Contract decision D — schema versioning

The HERMES_BASELINE.md row format does NOT bump `schema_version` 1→2 in this reframe. Per the specialist-board's amplified finding 1 (adversary + data-integrity + reversibility): a v1→v2 migration without an explicit migration path is TYPE-1 with high cost-if-wrong.

Instead: keep `schema_version=1`. The new columns (per-turn cost/latency, sample-size disclosure) become **additive** — appended to the row format, with NULL/empty values acceptable for nights where they can't be computed. Schema-version bumps are reserved for breaking changes, not additive ones. This is straight from the architect's finding 4 recommendation.

When per-turn cost attribution lands honestly (post-§5 resolution), the row format gains `_n_turns`, `_cost_per_turn_p50`, `_cost_per_turn_p95`, `_lat_per_turn_p95` as additional columns. `schema_version` stays at 1 throughout.

## §5 TBD — per-turn cost denominator (Codex review prompt)

Three options for cost-per-turn attribution. Each has honesty + complexity tradeoffs. Codex reviewer: pick one.

**Option A — Heuristic from session-level cost.** Compute `cost_per_turn = sessions.estimated_cost_usd / COUNT(messages WHERE role='user' AND session_id=s.id)`. Already-captured data; zero schema changes. Honesty: bounded error because session-level cost is real, denominator is a count, but assumes turns within a session have uniform cost — false when one turn does heavy retrieval and another is conversational.

**Option B — Langfuse trace span totals.** For each user-role message, find the corresponding Langfuse trace (keyed by `messages.session_id` + `messages.timestamp` proximity), sum `input_tokens + output_tokens` across all spans, multiply by per-model pricing. Captures the full cost driver including reasoning + tool calls + sub-agent invocations. Requires Langfuse to be running and trace-span query API to be stable. First-principles finding 3 endorses this as the honest path.

**Option C — Defer per-turn cost entirely.** Capture per-session cost only (the existing path) for G1's 7 nights. PF Runtime cutover gate uses session-level comparison. Per-turn cost becomes a post-G1 enhancement when billing-event-level instrumentation lands in PF Runtime. This is the architect's finding 2 recommendation if Hermes-side schema changes are out-of-scope for this window.

Recommendation: **B** if Langfuse is up, **C** otherwise. Avoid A unless both B and C are infeasible — the heuristic fiction shows up at 4.7.5 cutover comparisons.

## §6 TBD — latency definition (Codex review prompt)

Three latency interpretations, each measures something different. Codex reviewer: pick one and lock the definition in HERMES_BASELINE.md's Capture Environment section.

**(a) Wall-clock**: time between user message arriving and final assistant message being sent. Includes network round-trips, queue wait, tool execution. What the user perceives.

**(b) Agent-loop**: time the Hermes orchestrator code held the request — reasoning + retrieval + composition. Excludes tool wait time. What the runtime is responsible for.

**(c) Inference-only**: time the LLM was actively generating tokens. Excludes orchestration overhead, tool calls, retries.

For Phase 4.7.5 cutover comparison, the definition must be the SAME for Hermes and PF Runtime — otherwise the comparison is meaningless. Architect finding 3 made this load-bearing.

Recommendation: **(a) wall-clock**. Reasons: (i) the user-visible metric is what 4.7.5 should care about, (ii) Hermes's existing `sessions.started_at`/`ended_at` columns already capture session-level wall-clock, so per-turn variant is a derivation from `messages.timestamp` deltas, (iii) PF Runtime can match this without instrumentation choice — same SQLite write pattern.

## §7 TBD — sample-size confidence band (Codex review prompt)

If a night captures only 2 user-turns, what's the honest confidence band on cost p95? The architect's finding 5 noted N=35 (5 turns × 7 nights) is not comparable to N=150 (Promptfoo).

Options:

- **Bootstrapped 90% CI** per metric per night, computed at metric-time, written to the baseline row. Requires `_n_turns` in the row. Good honesty; small bash overhead per night.
- **Wide-band tolerance only**: declare cost/latency tolerance as ±50% (vs the original ±10%/±30%) to absorb low-N noise. Sloppy honesty; loses signal when N is high.
- **Rolling 7-night aggregate**: compute confidence bands on the union of 7 nights' turns rather than per-night. Gives the cleanest aggregate but loses per-night drift detection.

Recommendation: **bootstrapped 90% CI per night** + **rolling-7-night aggregate band** for the cutover gate. Two columns per metric: `_p50_band_low` and `_p50_band_high`. Cutover compares PF Runtime p50 against the rolling Hermes band, not against a fixed point estimate.
